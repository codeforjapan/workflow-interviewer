# gap-notes: 人事給与

**このファイルの役割**:
標準仕様書の「あるべきフロー」と自治体の現実の差分を記録する。
差分はそのままFDE提案の根拠になる。

---

## 標準仕様書が定めているが、現実に機能していないこと

### ギャップ1：会計年度任用職員の大幅増加により、正規職員と異なる処理が混在し業務負荷が増大している

**仕様書の記述**: 職員は「正規職員」（継続雇用）と「会計年度任用職員」（年度ごとの任用）の2区分で管理される

**現実に起きていること**:
- 令和2年度に「会計年度任用職員」制度が導入され、従来の非常勤職員がこちらに移行
- 会計年度任用職員は「会計年度（4月～3月）内での任用」のため、毎年3月末で任用終了→4月に新規任用という独特の処理が発生
- 給与計算担当者が「この人は正規職員か任用職員か」で処理方法が変わることに対応するため、複数の計算ロジックを保有する必要がある
- 同一の業務を行っていても給与体系・保険加入・昇進制度が大きく異なり、職員間の処遇格差が拡大している
- 人事管理システムが正規職員向けの設計のままで、任用職員の管理機能が不足しており、担当者が Excel で別途管理している
- 毎年3月末の一括終了・4月初の一括採用フローで、システムエラーやデータ漏落が起きやすく、給与振込遅延の原因になることがある
- 任用職員の「評価制度」をどう導入するかが自治体ごとに異なり、給与への反映方法も統一されていない

**FDE提案の根拠**:
仕様書は「2つの職員区分を管理する」と定めるが、具体的な給与体系・福利厚生・人事評価の処理フローは定めていない。
会計年度任用職員は全国の自治体で急増し（令和2年度時点で約60万人）、
現在の人事管理・給与計算システムでの対応が追いついていない状況。
FDE層で「会計年度任用職員の標準的な給与計算フロー」「評価制度の実装パターン」
「3月末～4月初の一括処理の標準手順」を定義することで、
自治体ごとの手作業と属人化を減らせる。

### ギャップ2：人事評価結果の給与への反映処理が手動で行われているケースが多い

**仕様書の記述**: 人事評価に基づいて、職員の給与（特に期末手当・勤勉手当）を決定する

**現実に起きていること**:
- 人事評価システムと給与計算システムが連携していない自治体が多く、
  評価結果を人事課が手動で給与計算システムに入力している
- 評価票（紙またはシステム）から「評価ランク」を手作業で集計し、
  給与計算システムへ「係数」を手入力するプロセスが存在する
- 評価結果の入力間違い（「優秀」と「良好」を見間違える）が月1回程度の給与計算で発生し、
  給与支払後に「支給額が間違っていた」と判明し、返金処理が発生する
- 人事評価で「要改善」判定の職員への給与反映（減額）が、対象者ごとに異なる計算ロジックで行われており、
  同じランクの職員でも減額幅がばらつく場合がある
- 給与計算と人事評価のサイクルが合致していない自治体が多く、
  「3月の評価確定」から「6月のボーナス支給」までに3か月のタイムラグがあり、
  その間に人事異動が発生するため「どの評価を適用するか」で問題が生じる

**FDE提案の根拠**:
仕様書は「評価に基づいて給与を決定する」と定めるが、「どの評価項目をどう給与に反映させるか」の具体的なロジックを定めていない。
評価結果から給与反映までの自動化を進めるには、
FDE層で「評価ランク→給与係数の変換テーブル」「給与計算フローへの評価データの自動連携仕様」を標準化する必要がある。
これにより手入力エラーが減り、職員も「評価結果が給与にどう反映されるか」が理解しやすくなる。

### ギャップ3：育児・介護・病気休職等の多様な休暇種別の管理が複雑で担当者の経験知に依存している

**仕様書の記述**: 職員の休暇（年次有給休暇、特別休暇、休職）をシステムで管理し、給与計算に反映する

**現実に起きていること**:
- 年次有給休暇だけでなく、育児休暇・介護休暇・病気休暇・特別休暇（結婚・喪失など）・産前産後休暇など、
  多くの休暇種別が存在し、各種別で「給与支給の有無」「手当の対象」が異なる
- 育児休暇は「給与なし」だが、育児手当が支給されるケース・されないケースが自治体で異なる
- 介護休暇の日数制限（「年10日まで」か「通算93日まで」か）が法令改正で変わり、
  古い設定のシステムでは対応できず、手作業で調整している自治体が多い
- 病気休暇と病気休職の区分（「90日までは有給、それ以降は無給」など）が複雑で、
  担当者が「この職員はいつから休職に切り替えるか」を手作業で判定している
- 給与計算システムに「休暇種別ごとの給与控除設定」が多数あり、
  一度も使われていない古い設定がシステムに残っている（メンテナンス負荷）
- 長期休暇中の職員が多い月の給与計算は複雑度が高く、
  担当者が「間違いがないか」と何度も確認する必要があり、給与支払が遅れる

**FDE提案の根拠**:
仕様書は「休暇を管理する」と定めるが、休暇種別ごとの給与処理ルールは法令と各自治体の条例に基づいており、
全国統一の仕様は困難。
しかし「一般的な休暇種別」（年次有給、育児、介護、病気、産前産後）の「標準的な給与控除ロジック」を
FDE層で定義し、複数ベンダーのシステムに実装させることで、
中小自治体の給与計算担当者が「判断に迷わずに済む」ようになる。
また、法令改正時にFDE層で標準を更新すれば、全自治体が一斉に対応可能になる。

### ギャップ4：定年延長（令和5年～）に伴う再任用・定年前再任用短時間勤務の処理が標準化されていない

**仕様書の記述**: 職員の採用・異動・退職の処理を標準化する

**現実に起きていること**:
- 令和5年度から「定年延長制度」が施行され、60歳定年が段階的に65歳定年に延長される
- 延長対象職員は「定年延長職員」として引き続き正規職員扱いだが、
  給与・昇進・人事評価が段階的に変わり、新しい人事管理ロジックが必要
- 「定年前再任用短時間勤務」制度が新設され、60代職員が「フルタイム→短時間勤務」に段階的にシフトできるようになったが、
  この人事区分に対応した給与計算・福利厚生の設定がシステムに未実装の自治体が多い
- 再任用職員（定年退職後の非常勤任用）と「定年延長職員」の2つの職員区分が並行して存在し、
  同じ年代でも処遇が異なる
- 定年延長に伴う「給与表の改定」（60歳以降の昇給ルール変更など）が必要だが、
  自治体ごとに対応方法が異なり、統一されていない
- 給与計算システムが「正規職員」「会計年度任用職員」の2区分を前提に設計されており、
  新しい「定年延長職員」「短時間勤務職員」の区分に対応していない

**FDE提案の根拠**:
定年延長制度は令和5年度に施行され、標準仕様書発行（令和5年）後の新制度であるため、
仕様書に組み込まれていない。
全国の自治体が「定年延長に対応した人事給与システムの設定」に悩んでおり、
ベンダーも対応版を開発途上。
FDE層で「定年延長職員の標準的な人事管理・給与計算フロー」「短時間勤務への移行シミュレーション機能」
「給与表の段階的改定方法」を標準化することで、
自治体ごとの試行錯誤を減らし、統一的な対応が実現できる。
また、この定年延長制度は全国の自治体が同時期に対応するため、
FDE層での標準化が各自治体の手作業を大幅に削減できる。

→ 関連法令: 地方公務員法 改正（令和3年）

## 標準仕様書が定めておらず、自治体が個別設計している領域

| 領域 | 内容 | 難易度 |
|---|---|---|
| 人事評価の実施方法 | 4段階評価か5段階か、相対評価か絶対評価か | 高 |
| 評価結果の給与反映率 | 「優秀」なら+5%、「要改善」なら-3%など | 高 |
| 会計年度任用職員の評価制度 | 評価対象か対象外か、給与に反映させるか | 高 |
| 定年延長職員の給与設定 | 61歳以降の昇給ルール、給与上限 | 高 |
| 短時間勤務への移行ルール | 自発的か組織方針か、給与減額比率 | 中 |
| 各種休暇の給与控除 | 育児休暇中の手当支給、介護休暇の扱い | 中 |
| 超過勤務手当の上限 | 「月45時間まで」か「100時間まで」か | 中 |
| 福利厚生の職員区分別適用 | 正規と任用職員で異なる制度 | 低 |

## 差分の構造的な意味

```
人事給与の「穴」は雇用形態の多様化と
法制度変化のスピードが、
システム開発のスピードを上回ることから生じる

  会計年度任用職員制度（令和2年）
    + 定年延長制度（令和5年）
    ↓ 複数の職員区分が共存
    ↓ 各区分で給与体系・評価方法が異なる
    ↓ 人事管理システムが「正規職員中心」で設計されているため、
      新しい区分への対応が不完全
    ↓ 担当者が複数の計算ロジックを手作業で並行管理
      ↑ ヒューマンエラー、給与支払遅延の原因に

特に小規模自治体では給与計算担当者が1～2名のため、
複雑な制度対応によって業務負荷が大きく増加している。

FDE層では、「複数職員区分の標準的な管理方法」
「法令改正時の対応テンプレート」を整備することで、
各自治体がシステムを更新する際の設計時間を短縮できる。

また「定年延長制度の給与表改定方法」など、
全国の自治体が同じ時期に同じ課題に直面しているため、
FDE層での標準化による効果が非常に大きい。
```
