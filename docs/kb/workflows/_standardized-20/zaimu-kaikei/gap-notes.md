# gap-notes: 財務会計

**このファイルの役割**:
標準仕様書の「あるべきフロー」と自治体の現実の差分を記録する。
差分はそのままFDE提案の根拠になる。

---

## 標準仕様書が定めているが、現実に機能していないこと

### ギャップ1：支出負担行為から支払までの複数の承認フローが紙で回っていて電子決裁と混在している

**仕様書の記述**: 支出負担行為→支出命令→支払という3段階の手続きをシステムで管理し、各段階で承認を取得する

**現実に起きていること**:
- 支出負担行為は財務会計システム内で電子決裁されるが、支出命令は別の様式を紙で出納室に手渡ししている自治体がほとんど
- 支出命令の複数承認段階（課長→部長→会計課長→出納室）が、紙での回覧で回され、各承認者の判子を押している
- 会計課がシステムと紙の両方を突き合わせて整合性をチェックしており、二重管理の手作業が発生している
- 紙で回る過程で支出命令書が積み上がり、「この支出命令書は誰のところにあるのか」と追跡できず、支払が数日遅れることがある
- 電子決裁と紙決裁が混在しているため、決定日時（「いつ承認されたのか」）の記録が曖昧になりやすい

**FDE提案の根拠**:
仕様書は「各段階での承認」を定めるがシステム化の範囲を明示していない。
支出負担行為から支払まで全段階で電子決裁を実装することは「DX目標」としては当然だが、
実装の難易度（庁舎内システム統合、稟議経由の複数課の承認ルール統一など）が高く、
多くの自治体が部分的に電子化したままになっている。
FDE層で「支出命令の電子決裁化の実装パターン」（小規模自治体向け、中規模向け、など）を
提示することで、紙と電子の混在を解消するロードマップが提供できる。

### ギャップ2：会計年度任用職員の増加で人件費処理の複雑さが増している

**仕様書の記述**: 職員給与（人件費）は予算科目として計上し、月次で支出処理する

**現実に起きていること**:
- 令和2年度に「会計年度任用職員」制度が始まり、従来の「非常勤職員（日給月給制）」と異なる処理体系が導入された
- 会計年度任用職員は「会計年度（4月～3月）内での任用」のため、
  毎年3月末で任用終了→4月に新規任用という手続きが発生し、
  給与計算システムでのデータリセット・更新が複雑化している
- 通常職員と会計年度任用職員の2つの給与体系が並行して運用されており、
  給与計算担当者が「この人は退職金対象か？」「給与水準は？」と確認するたびに混乱する
- 財務会計システムへの給与データ連携が、給与計算システム側の複雑な設定が必要で、
  バージョンアップのたびにエラーが出る
- 予算編成時に「会計年度任用職員の給与」を予算科目のどこに計上するかが、
  自治体ごと・部門ごとに異なり、決算段階で「科目の配置が間違っていた」と気づくケースが多い

**FDE提案の根拠**:
仕様書は「人件費は支出科目」と定めるだけで、通常職員と任用職員の併用時の処理フローを定めていない。
この複雑さは各自治体が自前で処理ルール（例：「給与システム→会計システムへの日次連携」「予算科目の自動配置ルール」）を開発している状況。
FDE層で「会計年度任用職員の人件費処理の標準フロー」を定義し、
複数ベンダーの給与システム・会計システムでの実装パターンを示すことで、
自治体ごとの手作業が減り、ヒューマンエラーも減る。

### ギャップ3：EBPM推進のために必要な事業別コスト情報が財務会計システムから取り出しにくい

**仕様書の記述**: 支出は予算科目（款・項・目）に分類して管理する

**現実に起きていること**:
- 予算科目は「機能的分類（総務費・民生費など）」に基づいており、
  各事業のコストを横断的に把握するのに向いていない
  （例：「保育施設への補助金」は民生費・児童福祉費だが、
  　「保育施設の施設整備」は土木費・公債費に分かれることがある）
- EBPMで「この事業にいくらコストがかかったか」を知るには、
  複数の予算科目にまたがるデータを手作業で集計する必要がある
- 事業別予算管理システムを導入している自治体でも、
  財務会計システムとの連携が不十分で、
  決算段階で「決算資料の数字」と「事業別コストの数字」が合わなくなっている
- 指標管理や行政評価に必要な「事業のコスト」データが、
  会計システムからは抽出しにくく、別システムまたは手作業で管理している

**FDE提案の根拠**:
予算科目は法定の分類体系であり、変更はできない。
しかし「事業単位でのコスト管理」は現代の行政経営（EBPM、経営目標管理など）の必須要件。
FDE層で「財務会計システムから事業別コストを抽出するための標準マッピング」
（「この予算科目が該当する事業はこれ」という対応表）を定義し、
複数ベンダーの会計システムでの実装を促すことで、
全自治体が簡単に事業別コスト情報にアクセスできるようになる。

### ギャップ4：決算審査のための資料作成に多大な手作業が残っている

**仕様書の記述**: 決算書、合計残高試算表、決算附属書類（各種調書・統計表）を作成し、監査委員へ提出

**現実に起きていること**:
- 決算附属書類（例：「債務負担行為の変更額調書」「基金の増減に関する調書」など）は
  ほぼ手作業（Excel や Word）で作成されている
- 会計システムからデータをエクスポート→Excel で加工→再度集計→Word で成形、という流れが
  数十種類の書類について繰り返される
- 監査委員から「この数字を詳しく見たい」と言われるたびに、
  似た資料を別途作成する（同じデータを複数形式で管理することになる）
- システムベンダーが「標準の決算書出力」を提供しているが、
  各自治体固有の出力様式（欄外に条例の条番号を記載、など）に合わせると
  カスタマイズが大量に必要になり、費用がかさむ
- 毎年の決算期間（通常6月30日の監査委員提出期限）が迫ると、
  会計課は手作業の調書作成に追われ、他業務が停滞する

**FDE提案の根拠**:
仕様書は「決算書を作成する」と定めるが、決算附属書類の形式・内容は自治体の条例に基づいており、
全国統一はできない。
しかし「決算書の基本構成」「必須の附属書類」は多くの自治体で共通している。
FDE層で「決算附属書類の標準的な調書テンプレート」を定義し、
複数ベンダーの会計システムが共通フォーマットで出力できるようにすれば、
決算期間の手作業を大幅に軽減できる。

## 標準仕様書が定めておらず、自治体が個別設計している領域

| 領域 | 内容 | 難易度 |
|---|---|---|
| 支出負担行為の承認権限と金額基準 | 「課長は500万まで、部長は1000万まで」など | 高 |
| 支出命令と監査委員への報告フロー | 全支出命令か、抽出報告か、事後報告か | 高 |
| 決算調整仕訳の処理フロー | 未払金・前払金の計上基準、決算調整項目の判定 | 中 |
| 事業別予算管理システムとの連携 | 予算科目から事業コストへのマッピング | 中 |
| 決算附属書類の作成・出力フロー | どの書類を誰が作成し、いつ提出するか | 中 |
| 支払遅延時の利息計算 | 年利何％、どの契約から対象か | 低 |
| 不納欠損の判定基準 | 何年未納なら欠損か、監査委員への報告 | 低 |

---

## 差分の構造的な意味

```
財務会計の「穴」は予算科目体系と事業管理のズレから生じる

  予算科目（法定分類：款・項・目）
    ↓ 機能別分類のため、事業別コストが把握しにくい
    ↓ 支出負担行為～支払までの承認が紙と電子の混在
    ↓ 会計年度任用職員の複雑な給与処理で重複管理
    ↓ 決算附属書類の大量手作業
      ↑ 結果として、決算から事業評価への反映が遅い

現代の行政は「決算」を実績確認だけでなく、
次年度予算編成とEBPM（エビデンスベースの施策立案）に活用する必要がある。
しかし現在の財務会計システムは「予算科目による予算管理」と
「事業別のコスト把握」の2つのニーズを完全には満たしていない。

FDE層では、既存の予算科目体系を変えずに、
「事業別コストの自動抽出」「決算附属書類の標準出力」
「支出命令の完全電子決裁化」といった機能強化を標準化することで、
自治体の意思決定をより迅速・正確にすることができる。

特に小規模自治体では予算・決算の専任者が少なく、
手作業負荷が大きいため、FDE層での標準化による業務軽減は
直結して自治体職員の働き方改革につながる。
```
