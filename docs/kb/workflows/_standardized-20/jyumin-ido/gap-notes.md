# gap-notes: 住民異動届

**このファイルの役割**:
標準仕様書の「あるべきフロー」と自治体の現実の差分を記録する。
差分はそのままFDE提案の根拠になる。

---

## 標準仕様書が定めているが、現実に機能していないこと

### ギャップ1：庁内他業務連携（仕様書7.2章）

**仕様書の記述**: 転入処理後、国保・介護保険等へ自動データ連携

**現実に起きていること**:
- 住基システムと給付管理システムが**日次バッチ**でしか連携しない自治体が多い
- 転入当日に児童手当を申請しようとすると、給付管理システムに住所が未反映
- 担当者間の電話確認という「人力API」で補完されている

**FDE提案の根拠**:
仕様書7.2章は「連携すること」を定めているが、**連携のタイミング**は定めていない。
リアルタイム連携かバッチ連携かは実装次第。
「標準準拠システムに移行したのに当日連携できない」状態は、
仕様書の趣旨に反する状態として改善提案できる。

→ 関連インシデント: `incident-catalog/INC-001-dv-cross-department.md`

### ギャップ2：支援措置フラグの他課連携

**仕様書の記述**: 支援措置フラグを住民基本台帳システムで管理（4.1節）

**現実に起きていること**:
- 住基システムのDVフラグは、**給付管理システムには連携されない**
- 支払通知書・決定通知書への住所印字制御は各課の個別判断
- フラグ情報を他課に伝えるのは係長間の電話のみ

**FDE提案の根拠**:
仕様書は「支援措置フラグの管理」を住基システムの機能として定義しているが、
「そのフラグを他システムに連携すること」は定義していない。
ここが制度設計上の「穴」であり、**標準化の次のフェーズで対応が必要な課題**として
デジタル庁に問題提起する材料になる。

→ 関連インシデント: `incident-catalog/INC-001-dv-cross-department.md`

### ギャップ3：関連手続き案内（標準オプション8.1）

**仕様書の記述**: 本人通知サービスは「標準オプション機能」（採用は任意）

**現実に起きていること**:
- 「関連手続き案内」（児童手当・介護保険等への案内）は仕様書に定義がない
- 担当者の個人知識・チェックリストの有無・自治体の文化に完全に依存
- 新人担当者は案内すること自体を知らない

**FDE提案の根拠**:
これは標準仕様書の**空白領域**。
「標準準拠システムに移行すれば解決する」問題ではなく、
「業務フロー設計とナレッジ整備が必要」な問題。
CfJが介入できる典型的な領域。

→ `knowledge-notes.md` の「関連手続き案内」セクションを参照

## 標準仕様書が定めておらず、自治体が個別設計している領域

これらは標準化後も各自治体で設計が必要な部分。
FDEとして入る際の「作業対象」リスト。

| 領域 | 内容 | 難易度 |
|---|---|---|
| 転入時の関連手続き案内フロー | 何を・誰が・どのタイミングで案内するか | 中 |
| DV案件の他課連絡プロセス | 連絡票・連絡先・タイミングの設計 | 高 |
| 繁忙期（3月末）の処理優先ルール | 入力が追いつかない場合の判断基準 | 中 |
| 外国人住民の対応手順 | 通訳手配・在留資格確認フロー | 高 |
| 会計年度任用職員の権限範囲 | どこまでできてどこから正職員か | 低 |

## 差分の構造的な意味

```
標準仕様書が定めること
  = システムが「できること」「できないこと」の定義
  = データ構造と機能の標準化

標準仕様書が定めないこと
  = 「誰が・いつ・どう判断するか」
  = 「制度間の連携をどう設計するか」
  = 「住民にどう説明するか」

     ↑ここがCfJの仕事領域
```

標準化が進むほど「システムが解決しない問題」が明確になる。
gap-notes.mdはその問題を記録するファイルである。

## ギャップ4：災害時の住所特例（住民異動業務との交差）

**通常の住民異動フロー**:
転入届を出した市区町村が、国保・介護保険・各種給付の所管市区町村になる。

**大規模災害発生時に起きること**:

### パターンA: 住民票を移さずに避難（みなし仮設・縁故避難）
- 被災地の市区町村に住民票を残したまま、他市区町村の民間賃貸住宅（みなし仮設）に長期避難
- 介護保険・国保の窓口は元の市区町村のまま → 避難先の市区町村で各種手続きができない
- 「住民票を移すと仮設住宅の利用資格がなくなる」と誤解して移さないケースがある
- 対策: 災害特例として「現在の居住地の市区町村でも手続き可能」にする措置の周知

### パターンB: 仮設住宅入居で住民票を移動した場合
- 仮設住宅の住所に転入届 → 元の被災地の市区町村の行政サービスが使えなくなる
- 被災地の市区町村が行う「地元住民向けの支援事業」の対象から外れるリスク
- 「住民票を移すべきかどうか」を窓口で相談されるが、正解がない（制度上の矛盾）

### パターンC: 被災により住民基本台帳自体が機能しない
- 大規模水害・地震で市区町村の庁舎・サーバーが被災
- 住民票データが消失または参照不能になる
- バックアップデータ（都道府県・国の保持分）からの復旧が必要
- 復旧期間中の窓口対応（住民票が確認できない状態での手続き）の判断基準が必要

**仕様書との関係**:
標準仕様書は「転入届に基づく情報更新」を定めているが、「届を出せない状況での住所認定」は定めていない。災害時の住所特例は厚労省・内閣府の通知で示されるため、その都度確認が必要。

→ 関連: `concepts/domicile.md`（住所・居住地の災害時特例）
