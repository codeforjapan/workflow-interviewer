# 共通庶務・典型業務フロー集

## このディレクトリの位置づけ

デジタル庁の**標準仕様書が存在しない業務**であるが、**全国の自治体で共通して行われる典型的な行政実務**のフロー集。

### 標準化対象業務との違い

| 視点 | `_standardized-20/` | `administrative-commons/` |
|---|---|---|
| **出典** | デジタル庁・各省の標準仕様書 | 全国自治体の一般的な実務運用 |
| **フロー記述** | `flow-standard.md`（仕様書に基づく「あるべきフロー」） | `flow-canonical.md`（典型例） |
| **位置づけ** | 「このフローが正しい」 | 「おそらくこうなっている。どこが崩れているか確認しよう」 |
| **ギャップの意味** | 仕様書と現実の離隔 | 典型例と個別運用の個差 |

### ファイル構成

各業務フォルダに以下の2ファイルを置く。

| ファイル | 内容 |
|---|---|
| `flow-canonical.md` | 全国的に見られる典型的なフロー（Mermaid図）と手続きの説明 |
| `gap-notes.md` | 典型例と実際の自治体運用との差分、暗黙知、改善の留意点 |

### なぜ典型フローを記録するのか

---

#### 背景：暗黙知と属人性

標準仕様書のない業務は、各自治体が「自分たちのやり方」を定めている。しかし多くの場合、そのやり方は：

- **文書化されていない**（「いつもこうやってる」で済ませている）
- **担当者に依存**している（ベテランは知っているが新人は知らない）
- **組織内でばらつく**（係ごと、年ごとに微妙に違う）

結果として：
- **引き継ぎが失敗する**（「前はどうやってたのか」が不明）
- **品質がばらつく**（同じ業務なのに処理内容が違う）
- **改善が進まない**（「今のやり方が正しいのか」判断できない）

#### 解決策：「典型例」を明示する

全国の自治体の実務を観察すると、同じ業務をほぼ同じ手順で行っている。その「典型的なやり方」を可視化することで：

- **暗黙知を形式知に**：「当たり前だと思ってたこと」が明示される
- **バラツキを認識**：「うちはここが違う」という気づき が生まれる
- **改善の起点**：「ここをこう変えたら？」という提案が出やすくなる

## 各フォルダの内容概観

### 照査回答（shokai-kaito）

**業務定義**: 上位機関・議員・他課・住民からの照会に対して、担当課が回答を起案し決裁を経て返答する業務

**典型フロー**: 照会受信 → 内容分類 → 回答案作成 → 確認・決裁 → 送付 → 記録

**主要なギャップ**:
- 口頭照会が記録されない
- 期限が暗黙知（議員照会の事実上の締切）
- 回答の前例参照が個人依存
- 横断的な照会の責任所在が不明

### 軽微発注・納品検査（hatchu-nohinkensa）

**業務定義**: 随意契約が可能な金額以下の物品購入・印刷・修繕等の発注と納品検査

**典型フロー**: 予算確認 → 見積取得 → 起案 → 発注 → 納品 → 検査 → 支払

**主要なギャップ**:
- 随意契約の分割発注（脱法的運用）
- 納品検査の形骸化
- 見積合わせの記録が残らない
- 発注から検査まで単一担当者（牽制機能なし）

### 外部会議ロジスティクス（kaigi-logi）

**業務定義**: 審議会・有識者会議・ケース会議（要保護児童対策地域協議会、地域ケア会議等）など、外部の委員や関係機関を招集して行う会議の運営・調整・記録業務

**典型フロー**: 開催趣旨・委員確定 → 日程調整 → 招集通知・資料送付 → 当日運営 → 議事録作成 → 決定事項の周知・公開

**主要なギャップ**:
- 欠席委員の意見反映・代理出席の規定が不明確
- 議事録確定プロセスが属人的（確定時期のばらつき）
- 利益相反（COI）管理が形式的
- ケース会議の招集基準・参加機関の範囲が不定

### 食品衛生監査・立入検査（shokuhin-esei-kansa）

**業務定義**: 食品衛生法に基づく飲食店・食品製造業等への立入検査および営業許可更新に係る審査業務

**典型フロー**: 対象施設リスト整備 → 年間計画立案 → 立入実施 → 指導事項記録 → 改善指導 → 是正確認 → 記録保存

**主要なギャップ**:
- 指導事項の追跡管理が紙・担当者依存
- 是正確認の実施率・タイミングがばらつく
- 許可台帳と実態のズレ（廃業未届け施設の残存）
- 食中毒発生時の対応フローとの接続が弱い

### 消防査察・立入検査（shobo-kensa）

**業務定義**: 消防法に基づく防火対象物への立入検査。違反是正・命令・公表を含む

**典型フロー**: 対象選定（リスク評価）→ 査察計画 → 立入実施 → 違反事項確認 → 改善命令・指導 → 是正確認 → 命令解除・記録

**主要なギャップ**:
- 査察対象の網羅性（未把握施設・用途変更未届け）
- 是正命令後の追跡が属人的で漏れやすい
- 命令・公表の判断基準が不明確（担当によるばらつき）
- 消防本部と市役所の情報共有が不十分な場合がある

### 補助金管理（自治会・まちづくり団体）（hojokin-kanri）

**業務定義**: 自治会・町内会・まちづくり団体・NPO等を対象とした補助金の交付から実績確認・精算までの管理業務

**典型フロー**: 公募・申請受付 → 要件審査・採択 → 交付決定通知 → 事業実施（団体側）→ 実績報告受付 → 精算審査 → 確定・追加返還対応 → 効果測定・記録

**主要なギャップ**:
- 実績報告の形骸化（実施の有無確認のみで内容精査されない）
- 補助対象経費の解釈が担当者によりばらつく
- 補助金依存の固定化（同一団体への繰り返し交付の検証がない）
- 効果測定の基準がなく、廃止・見直しの判断根拠が作れない

## 使い方

### 実務担当者向け

新しい業務に担当が代わったとき、またはプロセス見直しを検討するときに：

1. `flow-canonical.md` を読んで「全国的な標準的なやり方」を理解する
2. `gap-notes.md` でよくある問題・陥りやすいポイントを確認する
3. 自分たちの運用と比較して「うちはどこが違うのか」を明確にする

### 業務改善・システム企画向け

業務改善やシステム導入を検討するときに：

1. `gap-notes.md` の「典型的な問題」をリストアップする
2. 自治体に現地ヒアリングしてギャップを確認する
3. 「これは典型的な困りごとなのか、この自治体固有の問題なのか」を区別する
4. 典型的な問題であれば、業務改善やシステム機能で共通的に解決できる
