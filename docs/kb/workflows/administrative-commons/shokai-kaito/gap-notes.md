# gap-notes: 照査回答

**このファイルの役割**:
「全国的な典型例」と実際の自治体の運用の差分を記録する。
これらの差分は、業務改善やシステム企画の根拠として活用される。

---

## 典型例と現実の主要なギャップ

### ギャップ1：口頭照査が記録されない

**典型例で想定している運用**:
- 照査受信 → 内容記録 → 回答案作成 → 決裁 → 送付 → 文書保存
- すべての照査が「受付記録」を通じて組織で把握可能

**現実に起きていること**:
- 「電話がかかってきて、その場で答えた」→ 記録がない
- 「上司に『この件どう答えたらいい？』と聞かれて、その場で相談した」→ 記録がない
- 「メールで個別に『こういうことですよ』と返信した」→ 受付台帳に載らない
- 結果、同じ照査が違う答えで返ってきたり、「あの時は何と言ったのか」が思い出せない

**なぜこうなる**:
- 「簡単な照査は記録しなくていいだろう」という判断のばらつき
- 電話受付時に「記録する時間がない」
- 受付台帳をつけるルール自体が浸透していない

**FDE/業務改善の観点**:
- 「どの程度の照査から記録するか」の明確な基準が必要
- 電話受付時の「簡易記録フォーム」（紙またはスマートフォン）の用意
- 受付記録を入力する時間をワークフローに組み込む

---

### ギャップ2：期限が暗黙知

**典型例で想定している運用**:
- 照査の種類ごとに「標準的な回答期限」がある
- 議員照査は緊急扱い（議会日までに）
- 上位機関照査は指定期限に従う

**現実に起きていること**:
- 「急ぎ」と言われても、実は議会が明後日だと気づかない新人
- 「これは時間かかりそう」と判断して3週間放置 → あとで議員に怒られる
- 「いつまでに？」と聞き返すのが失礼だと思い込んでいる
- 関係課への照会をした後、その返答を待っている間に期限が過ぎる

**なぜこうなる**:
- 「議員照査は特別」「上位機関からは急ぐ」という常識が新人に共有されていない
- 庁内スケジュール（議会日・定例会日）が意識されていない
- 「この照査の重要度」の判定が属人的

**FDE/業務改善の観点**:
- 照査受付時に「回答期限の判定フロー」を明確化
  - 発信者別の目安（議員→ ◎日まで / 県 → 指定期限 / 他課 → ◎営業日 等）
  - 議会日や定例会日との関連を自動で表示する
- 新人・異動者向けの「照査対応のポイント研修」の定期実施
- 進捗管理ツール（未回答照査の一覧・期限警告）

---

### ギャップ3：回答の前例参照が個人依存

**典型例で想定している運用**:
- 過去に同じような照査があれば、その回答を参考にする
- 同じ内容なら統一した見解を保つ

**現実に起きていること**:
- 「去年、同じような質問をもらった気がするんだけど…」とベテランに聞く
- ベテランが「◎年のあの件ですね」と思い出す（思い出さないこともある）
- 文書管理システムに保存されているが、検索方法を知らない / タイトルからは内容が推測しにくい
- 違う見解で回答してしまい、後から「去年は違う答えをした」と指摘されて混乱
- 部署異動でベテランが不在になると、「あの照査どう答えたのか誰も知らない」

**なぜこうなる**:
- 過去の照査・回答が体系的に分類されていない（保存されているが「探しづらい」）
- 「同じテーマの照査の一覧」が視覚化されていない
- 個人の経験に頼る運用になっている

**FDE/業務改善の観点**:
- 照査・回答を「テーマ別インデックス」で管理
  - 例：「社会福祉 > 生活保護 > 生活困難者への支援」という階層分類
  - 新しい照査を受けたときに「このテーマについては過去こんな回答をしている」が一覧表示される
- 定期的な「見解統一ミーティング」（同じテーマで矛盾した回答が出ていないか確認）
- デジタル庁標準の「FAQ」機能（頻出照査を共有可能なフォーマットで管理）

---

### ギャップ4：横断的な照査の責任所在が不明

**典型例で想定している運用**:
- 複数課にまたがる照査 → 関係課に相談 → 見解をまとめて回答
- どの課がまとめ役かは、照査内容によって決まる（自動的に判定される想定）

**現実に起きていること**:
- 「この照査は誰が回答すべき？」が最初からわからない
- 保健と福祉、両方に関わる質問 → どちらが主に答えるのか曖昧
- 「福祉課で回答します」と言っていたのに、実は保健課の領域だった → 誰かが気づいて協議開始
- 関係課への相談メールを出しても、返答が遅い / 返答がない
- 回答期限が迫ってきたのに、関係課との「見解統一」がまだ終わっていない
- 最後は「急ぐから各課の見解をそのまま並べて送信」→ 一貫性を欠いた回答に

**なぜこうなる**:
- 「どのテーマは誰の領域か」の責任分担が明確になっていない
- 複雑な照査ほど、庁内調整に時間がかかるが、それを見越した期限管理がない
- 関係課への依頼フロー（誰に、いつまでに、何を聞くのか）が標準化されていない

**FDE/業務改善の観点**:
- **照査の「主担当課」「副担当課」を自動判定する仕組み**
  - テーマごとに主担当課を決めておき、照査の内容キーワードから自動ルーティング
  - または、受け付け時に受け手が「この照査の主テーマは何か」を選択入力
- **「関係課協議期限」を設定**
  - 最終回答期限の3〜5日前に、関係課からの回答締め切りを設定
  - それ以降は「見解統一」に充てる
- **横断的テーマのリポジトリ**
  - 「社会福祉 × 税務」「都市計画 × 防災」といった、複数課が関わるテーマを事前に整理
  - そのテーマについて「市の統一見解」を定期的に確認・更新する

## 典型的な問題シーン

### シーン1：議員から「何か知ってるか」と電話がかかってきた場合

```
【現場での判断】
- 「知ってるか」ということなので、わかる範囲で答えればいい？
- 「正式な回答はまた後で送ります」でいい？
- この電話内容を報告すべき？

【典型例では】
→ 「わかりました。正式な見解を確認の上、文書で送付させていただきます」
→ 電話内容を上司に報告・記録
→ 後日、正式な文書回答を送付

【現実に起きる失敗】
→ その場で「たぶんこうです」と答えた
→ 記録がない
→ 後日、異なる見解で文書回答を送った
→ 「さっき電話で違うことを言ったじゃないか」と怒られる
```

---

### シーン2：「この件は何課が答えるべき？」という局面

```
【典型例では】
→ 「これは福祉課が主で、保健課に協力してもらう件ですね」と判定
→ 福祉課で回答案作成、保健課に確認
→ 一貫した回答を送付

【現実に起きる流れ】
→ 最初の受け手が「どちらが主かわからない」で判断停止
→ 上司に相談 → 上司も「どちらかな」→ とりあえず福祉課に回答させる
→ 福祉課が書いた回答を見て「あ、これ保健の領域も含んでる」と気づく
→ 保健課に協力依頼 → 返答が遅い
→ 期限が間に合わず、「福祉課の案のまま出す」（不完全）

→ 後で「ここの説明は間違ってる。保健では…」と指摘される
```

---

### シーン3：「回答の根拠は？」と聞かれた場合

```
【典型例では】
→ 「◎◎法 第▲条、および本市△△条例に基づいて…」と明確に説明
→ 必要に応じて過去の判例を示す

【現実に起きる流れ】
→ 「えっと、確か◎◎法だったと思うんですけど…」
→ 「法令を確認してからお返事します」と保留
→ でも、確認結果を明確に記録していない
→ 後で同じ質問をされて、また同じように「確認します」と時間がかかる
```

## 業務改善の入口

### すぐに取り組める施策

1. **受付台帳の実装**
   - 簡潔なフォーム（日付 / 発信者 / 内容 / 期限 / 回答予定者）
   - 紙でもExcelでも、まず「記録する」習慣をつける

2. **議員照査のチェックリスト**
   - 「議員からの照査を受けたら」という新人向けの対応フロー図
   - 「まず上司に報告」「期限は議会日までが鉄則」という明記

3. **過去回答の「テーマ分類」**
   - 文書管理システムに入っている過去の回答を、テーマ別に整理し直す
   - 新人が「社会福祉」「税務」といった大テーマで検索可能に

### 中期的な改善（システム企画の観点）

- 照査管理システム（受付 → 期限管理 → 決裁 → 記録）の整備
- AI による「回答テーマの自動判定」（主担当課の候補をシステムが提案）
- 定期的なテーマ別「見解統一レビュー」のプロセス化

## 他業務との関連

照査回答は、以下の業務と密接に関連している。

| 関連業務 | 連携ポイント |
|---|---|
| 文書起案・決裁回覧（administrative-commons） | 重要な照査の回答は、組織的な決裁フローを通る |
| 会議ロジスティクス（administrative-commons） | 横断的な照査の「見解統一」を定例会議で行う |
| 国民健康保険（workflows/_standardized-20） | 国保制度についての照査が頻出（保険料算定根拠等） |
| 生活保護（workflows/_standardized-20） | 生活保護関連の照査（特に議員から） |
