# バックオフィスジャーニー

## このディレクトリの位置づけ

行政組織が**「年度という時間軸」を中心に経験する内部業務の全体像**を示す置き場所。

住民ジャーニーが「ひとつのライフイベント」を軸にするのに対して、バックオフィスジャーニーは「年度サイクル」「議会対応」「人事異動」といった**組織が毎年繰り返す活動**に焦点を当てる。

たとえば「年度予算編成」という1つのサイクルに対して、実際に組織内で何が起きるか、どの課が何をするのか、どこで穴が生まれやすいかを記録する。

### 住民ジャーニー との違い

| 軸 | 住民ジャーニー | バックオフィスジャーニー |
|---|---|---|
| **対象** | 住民の人生イベント（引越し、出産、就職など） | 行政組織の年度サイクル |
| **起点** | 「住民に何かが起きる」 | 「暦が進む・季節が来る」 |
| **視点** | 住民が複数の課を巡る | 複数の課が共通して動く |
| **知見の源** | 个別 Case の経験 | 年度を通した時間軸の経験 |
| **ベテランが持つ暗黙知** | 「この状況で何が漏れやすいか」 | 「この時期に何をしておくとリスクを避けられるか」 |

### workflows/ や incident-catalog/ との違い

| 層 | ファイル置き場 | 視点 | わかること |
|---|---|---|---|
| **journeys/** (住民) | `journeys/` | 住民の立場 | 「ひとつのイベントで、どんな手続きが連鎖するか」 |
| **journeys/backoffice/** (組織) | `journeys/backoffice/` | 組織の立場（時間軸） | 「この時期に、全課で何が起きるか」「業務間で何が落ちるか」 |
| **workflows/** | `workflows/` | 担当者の立場 | 「この業務の正解フロー・判断基準・暗黙知は何か」 |
| **incident-catalog/** | `incident-catalog/` | 組織の立場（リスク） | 「この穴が放置されるとどんなインシデントが起きるか」 |

各層は互いを参照するが、**単独でも読める**ように設計されている。

---

## バックオフィスジャーニーの定義

**バックオフィスジャーニー** = 行政組織の**年度という時間軸を中心に**、内部業務がどのように連鎖・集中・重複して起きるのかを、月単位・週単位で示した業務構造図。

### なぜ可視化するのか

ベテラン職員が持つ「今の時期に何をしておけばリスクを避けられるか」という暗黙知は、この**時間軸上の業務構造**から来ている。

- 「3月末は支出が集中するから、検査がブレてはいけない」
- 「年度末の人事異動の時期は、継続事業の引き継ぎが曖昧になりやすい」
- 「予算編成の根拠がブラックボックスになると、ベテランが異動した途端に『なぜこの額？』がわからなくなる」

こうした知見を**文書化・時系列化する**ことで、新人・異動者・研究者が「なぜその業務がその時期に必要か」を理解でき、また組織全体で予防的な対応をとれるようになる。

## Oコード：バックオフィスサイクルの分類

住民ジャーニーが「L コード」（L = Lifecycle）を使うのに対して、バックオフィスジャーニーは「**O コード**」（O = Organizational cycle）を使う。

| O コード | サイクル名 | 説明 | ファイル |
|---|---|---|---|
| **O1** | **年度サイクル** | 予算編成〜予算執行〜決算。4月〜翌3月の全体構造 | [fiscal-year-cycle.md](./fiscal-year-cycle.md) |
| **O2** | **議会対応サイクル** | 定例会4回+臨時会。議案提出から閉会までの約2か月間の繰り返し | [council-response-cycle.md](./council-response-cycle.md) |
| **O3** | **人事異動サイクル** | 3月内示〜4月着任。年度交代に伴う配置変更と引き継ぎ | [staff-rotation-cycle.md](./staff-rotation-cycle.md) |
| **O4** | **監査・検査対応サイクル** | 決算監査、定期監査、行政評価への対応 | [audit-cycle.md](./audit-cycle.md) |
| **O5** | **災害対応サイクル** | 発災〜緊急対応〜復旧・復興。**非定期発動型**（発災時のみ起動）。O1と並行して動く | [disaster-response-cycle.md](./disaster-response-cycle.md) |

### O コード の読み方

各バックオフィスジャーニーは以下の要素を含む：

| 要素 | 役割 |
|---|---|
| **YAMLヘッダー** | `backoffice_cycle`（O コード）と関連する `workflows/` への参照 |
| **全体像** | テキスト図で「1年間（または1サイクル）で何が起きるか」を時系列で表現 |
| **業務連鎖マトリクス** | 各月の主要業務・関係課・締切・リスクを表で整理 |
| **穴の連鎖マップ** | この時期に起きやすい穴と、その連鎖先、対応する `incident-catalog/` ファイルへの参照 |
| **担当者向けチェックフロー** | 第一線の職員が使える確認リスト（時期ごと） |
| **住民ジャーニーとのクロス参照** | 「この時期に、どの住民ジャーニーと交差しやすいか」（例：3月末の転入集中 = O1 × L5） |

## 住民ジャーニー × バックオフィスジャーニー の交点

バックオフィスの年度サイクルと、住民のライフイベントが同時に起きるときが、**リスクが最も高い時期**。

### 代表的な交点

| 時期 | O コード | 住民ジャーニー | 何が起きるか | リスク |
|---|---|---|---|---|
| **3月末〜4月初** | O1 × O3 | L5（転入） | 年度末の支出集中＋人事異動＋転入ラッシュが同時発生 | 引き継ぎ不良、転入対応ミス、保育等の需要急増 |
| **3月末** | O1 | L2（保育入所） | 年度末決算＋保育入所決定 | 予算執行と人数確定が同時、待機児童対応が迫る |
| **3月末** | O1 | L7（介護保険） | 年度末決算＋介護保険年度更新 | 新年度の介護認定書発送が4月1日に間に合わない |
| **9月** | O2 | O1 | 9月議会（補正予算）と予算編成開始 | 前年度の課題と新年度の構想が同時進行 |
| **発災時（不定期）** | O5 × O1 | L13（被災・生活再建） | 通常の年度業務が継続する中で災害対応が同時発動 | 職員疲弊・通常業務の劣化・応援職員の指揮混乱 |
| **発災時（不定期）** | O5 | L8（おくやみ） | 災害死亡者の遺族手続きと災害弔慰金申請が並走 | おくやみと生活再建支援の窓口が分断される |
| **発災後〜数か月** | O5 | L12（困窮） | 被災による収入喪失・住宅損失から生活保護申請が増加 | 被災者の生活保護申請を「現在地保護」で受け付ける体制の確認 |

## 新規バックオフィスジャーニーの追加方法

新しいサイクルのジャーニーを追加する場合は、以下を参照してください。

→ [../templates/journey-template.md](../templates/journey-template.md)

テンプレートは、YAMLヘッダーの書き方から穴の連鎖マップの論理まで、すべての要素を示している。
自治体担当者・支援者・研究者からのコントリビューションを歓迎します。
