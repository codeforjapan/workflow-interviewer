# 住民ジャーニー

## このディレクトリの位置づけ

住民が**「ひとつのライフイベント」で経験する行政接点の全体像**を示す置き場所。

たとえば「引越し」「出産・育児」「結婚・離婚」「相続」といった人生のイベントに対して、実際に住民がたどる複数の行政手続きと、その業務間の関連性・穴・連鎖リスクを記録する。

### workflows/ や incident-catalog/ との違い

| 層 | ファイル置き場 | 視点 | わかること |
|---|---|---|---|
| **journeys/** | このディレクトリ | 住民の立場 | 「ひとつのイベントで、どんな手続きが連鎖するか」「業務間で何が落ちるか」 |
| **workflows/** | `workflows/` | 担当者の立場 | 「この業務の正解フロー・判断基準・暗黙知は何か」 |
| **incident-catalog/** | `incident-catalog/` | 組織の立場 | 「この穴が放置されるとどんなインシデントが起きるか」「どう予防するか」 |

各層は互いを参照するが、**単独でも読める**ように設計されている。

## ジャーニーファイルの共通構成

各ジャーニー（`*.md`）は以下の要素を含む。

| 要素 | 役割 |
|---|---|
| **YAMLヘッダー** | `psid_lifecycle`（ライフイベントID）と `psid_services`（関連サービスカテゴリ）を記載 |
| **全体像** | テキスト図で「何が起きて、誰が何をするか」を時系列で表現 |
| **業務連鎖マトリクス** | 対象者・期限・窓口・案内責任を表で整理（**漏れると何が起きるか**が目印） |
| **穴の連鎖マップ** | インシデントが発生する起点と連鎖先、対応する `incident-catalog/` ファイルへの参照 |
| **担当者向けチェックフロー** | 第一線の職員が使える確認リスト（**ここから読み始めることを推奨**） |
| **カスタマイズポイント** | 自治体がForkする際に変更が必要な箇所（窓口配置・システム連携・案内票様式など） |
| **関連ジャーニー** | このジャーニーと連鎖しやすい他の人生イベント |

## ジャーニーの読み方

**担当者は「チェックフロー」から読み始めることを推奨する。**

その理由：
1. チェックフローは**「今、眼の前にいる住民のために何をするか」という即時性**に焦点を当てている
2. そこから参照として `workflows/` で詳細フローを確認する
3. リスクを理解したら `incident-catalog/` で予防策を学ぶ

つまり、最初から全体を理解する必要はない。**その瞬間に必要な確認項目だけを使える**設計になっている。

## 新規ジャーニーの追加方法

新しいライフイベントのジャーニーを追加する場合は、以下を参照してください。

→ [templates/journey-template.md](../templates/journey-template.md)

テンプレートは、YAMLヘッダーの書き方から穴の連鎖マップの論理まで、すべての要素を示している。
自治体担当者・支援者・研究者からのコントリビューションを歓迎します。
